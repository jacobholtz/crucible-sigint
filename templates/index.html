CRUCIBLE SIGINT — ANALYST GUIDE

WHAT IS CRUCIBLE SIGINT?

CRUCIBLE is a passive OSINT infrastructure fingerprinting engine for mapping malicious infrastructure clusters from a single confirmed-bad seed. It supports investigation of any adversary infrastructure — nation-state C2, ransomware affiliate portals, commodity malware loaders, phishing kit clusters, brand-impersonation campaigns, and more.

▸ METHODOLOGY CREDIT — RYAN McDONALD
"Fingerprinting Malicious Infrastructure Using Free Resources"
Principal Security Engineer | USMC 0341  ·  Published LinkedIn, May 2026
Ryan's published methodology demonstrated — using only free passive sources (crt.sh, urlscan.io, DNS, WHOIS, manual JS inspection) — how a single confirmed-bad domain can map an entire adversary infrastructure cluster, identify reused build artifacts, and reveal layered brand/operational structure.
CRUCIBLE automates that methodology and extends it. The investigative framework is Ryan's work. This tool is the automation layer built on top of it.

Crucible automates the passive pivot Ryan performed manually — certificate transparency, DNS, IP/ASN intelligence, WHOIS/RDAP, and JS bundle scanning — and extends it with urlscan.io corroboration, cert issuance timelines, ThreatFox / OTX cross-reference, platform pivots (favicon · body · JARM · NS), and threat-actor pattern matching. It is a pivot platform, not a verdict engine — every panel is a fact surface.

All API calls are server-side (Python/FastAPI). Zero CORS issues. No browser restrictions. Core sources require no API keys; advanced sources (Shodan, VirusTotal, AlienVault OTX, abuse.ch, Censys) are optional and configured in SETTINGS.

DISCOVERY STAGES

Each stage feeds an independent panel. Stages run in parallel where possible; a failure in one stage doesn't block the rest. The scanbar shows live progress.

  • Cert Transparency (Cert Spotter → certkit.io → crt.sh → Censys): Queries CT logs for every TLS cert ever issued to the seed and its subdomains. Cert Spotter is the primary; the others are fallbacks so a crt.sh outage doesn't break discovery. Surfaces sister domains sharing cert infrastructure and flags admin / staging / internal subdomain patterns that often expose the operator layer.
  • DNS Resolution: All record types (A, AAAA, MX, NS, TXT, CNAME, SOA) via Google DNS-over-HTTPS. Reveals nameserver patterns, MX infrastructure, SPF records, and parking indicators.
  • IP Intelligence: Enriches resolved IPs with ISP, ASN, country, and hosting-type detection. Surfaces bulletproof hosting, CDN/proxy masking (Cloudflare, AWS Global Accelerator, Fastly), and hosting ASNs over-represented in adversary infrastructure.
  • RDAP Registration: Registrar, creation date, status flags, nameservers. Surfaces registrars over-represented in malicious registrations and fresh domains (<90 days).
  • urlscan.io Corroboration: Historical scans, additional IPs/ASNs from browser-based scanning. High scan count = well-known bad domain.
  • Cert Issuance Timeline: Time-series of cert issuance across the cluster. Bursts of simultaneous issuance = automated kit spin-up; staggered issuance = organic growth.
  • Neighbor Domains: Reverse-IP lookup. Other domains hosted on the same IP — often co-located adversary infrastructure. Click any row to pivot.
  • ThreatFox / OTX Cross-Reference: Checks abuse.ch ThreatFox and AlienVault OTX for documented IOCs on the seed or its IPs. Direct hits are decisive evidence.
  • Platform Pivots (Favicon · Body · JARM · NS): Hash-based pivots — MurmurHash3 favicon, normalized body hash, JARM TLS fingerprint, and nameserver clustering. Finds infrastructure cousins that don't share certs or IPs but share build artifacts.
  • Threat Actor Attribution: Pattern-match the seed and its cluster against catalogued actor profiles (registrar choice + hosting + naming + JS pattern combinations).

HOW TO READ THE PANELS

Crucible is a pivot platform, not a verdict engine. Every panel is a fact surface — each row is a pivot point you can click or copy to follow the operator's infrastructure. Treat findings as evidence, not as a risk score.

Bulletproof / abuse-tolerant registrars

Over-represented in malicious registrations and slow to respond to abuse reports. Common examples: gname.com, NiceNIC, Reg.ru, and (for some campaigns) NameSilo. Surfacing one in the RDAP panel is a strong tell when paired with other indicators in the same cluster.

Mainstream registrars

GoDaddy, Namecheap, Hostinger, Tucows — common on both legitimate and abused domains. Informative when stacked with other tells; not definitive on their own.

Privacy-proxy registrars

PrivacyGuardian, Whoisguard, Hide.ms — used to hide attribution. Many legitimate domains use these too, so treat as moderate context.

EFFECTIVE INVESTIGATION WORKFLOW

  • Start with a confirmed-bad seed. One domain from a sandbox report, threat-feed hit, incident response, or law-enforcement disclosure. The pivot works from that single seed.
  • Let CT build the cluster. The wildcard query (%.domain.com) surfaces all subdomains and often reveals sister infrastructure sharing the same cert chain. Adversary stables of dozens-to-hundreds of domains routinely emerge from a single seed this way.
  • Check operator-layer subdomains immediately. Anything matching admin / panel / staging / internal / manage / ops patterns is likely a back-office surface. Those are the most sensitive findings — they reveal the operator layer, not the victim-facing frontend.
  • Look for layered brand structure. Adversaries commonly run multiple branded surfaces (lure → landing → C2; recruitment → platform; affiliate portal → leak site). Finding only one layer = finding part of the operation. Pivot via shared NS, favicon, JARM, or cert chain to find the rest.
  • Use platform pivots when CT/IP pivots are exhausted. Favicon hash, normalized body hash, JARM TLS fingerprint, and shared nameservers find infrastructure cousins that rotate certs and IPs but keep their build artifacts.
  • Defang before sharing. Use the DEFANG toggle on all exports. Standard TLP convention: dots → [.], :// → [://].

MODES

  • Standard: Full discovery pipeline across every panel. Use for initial investigation of any suspected malicious domain or IP.
  • Investigator: Verbose output, raw API responses in JSON export. Use when you need full evidence chain for a report or LEA referral.
  • Phishing / Brand Abuse: Point at YOUR domain to find lookalikes in cert transparency. Returns risk score, live DNS check, age of each lookalike, and DNStwist permutations marked LIVE / ACTIVE.
  • Cert Intelligence: Direct CT query with wildcard support. Use for subdomain enumeration, CA distribution, or historical cert pattern analysis.
  • Bulk IOC: Paste up to 50 domains/IPs. Enriches each with DNS, IP intel, and RDAP. Exports SIEM-ready CSV.
  • Revalidation: Register domains to a watchlist and re-check them on a schedule (hours between checks is configurable per-domain). Surfaces infrastructure decay (NXDOMAIN, IP/ASN changes, takedowns) and reactivation. Pull a DECAY REPORT or RECENT ALERTS at any time. Must be enabled in SETTINGS first.
  • Settings: Configure optional API keys (Shodan, VirusTotal, AlienVault OTX, abuse.ch, Censys, Cert Spotter), toggle CT sources and their priority order, register pluggable CT providers (scantower.io) via URL template + bearer key, and enable the revalidation scheduler.

PHISHING / BRAND ABUSE DEMOS — CLICK TO LOAD

These are legitimate brand domains — use them as seeds in the Phishing/Brand Abuse mode to find real lookalikes attackers have registered in cert transparency. The tool searches crt.sh for every cert containing the brand name and scores each lookalike by risk. These brands are documented as the most heavily impersonated globally.

paypal.com
TARGET: FINANCIAL / PAYMENT
Consistently the most impersonated financial brand. Expect hundreds of lookalikes: paypal-secure.com, paypallogin.net, pay-pal-verify.com. Common patterns: -secure, -verify, -login suffixes. Most use Let's Encrypt DV certs to appear legitimate. High CT visibility.
HUNDREDS OF LOOKALIKES CREDENTIAL HARVEST
coinbase.com
TARGET: CRYPTO EXCHANGE
Heavy phishing target. Attackers register clusters of lookalikes before market volatility events to harvest credentials. Common: coinbase-login.com, coinbasewallet.net, coinbase-support.org. Lookalikes often appear in CT in bursts coordinated with broader social-engineering campaigns targeting exchange users.
CLUSTER REGISTRATION CRYPTO CREDENTIALS
microsoft.com
TARGET: ENTERPRISE / M365
Ranked #1 or #2 most impersonated brand globally (Check Point Q4 2025). M365 credential theft = full corporate email access. AiTM (adversary-in-the-middle) phishing kits like Tycoon 2FA and Rockstar 2FA specifically target Microsoft auth flows. Expect MX records pointing at phishing infrastructure.
#1 IMPERSONATED AITM KITS
metamask.io
TARGET: CRYPTO WALLET
Wallet credential and seed phrase harvesting. Documented homoglyph attacks using lookalike Unicode characters (Japanese Hiragana ん) in domain names since late 2024. Also impersonated via fake Chrome extensions. Seed phrase capture = complete wallet drain with no recovery possible.
SEED PHRASE HARVEST HOMOGLYPH ATTACKS
amazon.com
TARGET: E-COMMERCE / CLOUD
Top-3 impersonated brand. Two distinct attack surfaces: consumer credential + payment card theft (fake order confirmation, account suspension), and AWS credential theft targeting developers (fake console login pages). Both show high CT lookalike volume with -login, -secure, -account patterns.
DUAL ATTACK SURFACE HIGH CT VOLUME
apple.com
TARGET: CONSUMER / APPLEID
iCloud credential harvesting via fake storage-full alerts. "Review and sign" phishing via fake AppleID pages. High-value target because AppleID = iCloud photos, Find My (device tracking), payment cards, and often reused password. Expect apple-id.com, icloud-login.net, appleid-verify.com patterns.
ICLOUD HARVEST STORAGE ALERT LURE

TECHNICAL NOTES

  • Core sources (free, no auth): Cert Spotter, certkit.io, crt.sh (CT logs), freeipapi.com (IP intel), dns.google (DoH), rdap.org (registration), urlscan.io (historical scans).
  • Optional sources (free key, configured in SETTINGS): Shodan (open ports / banners), VirusTotal (passive DNS, detections), AlienVault OTX (IOC pulses), abuse.ch ThreatFox / URLHaus / MalwareBazaar (single Auth-Key), Censys (substring brand search).
  • Pluggable CT providers: scantower.io is configured with an API key + URL template containing {q}. CRUCIBLE GETs that URL with Authorization: Bearer <key> and extracts domain-like names from the JSON.
  • CT source order: Cert Spotter is primary, certkit.io and crt.sh are fallbacks, Censys runs when keyed. A crt.sh outage no longer breaks discovery.
  • crt.sh flakiness: Single Postgres instance, goes down under load. If all CT sources fail simultaneously, retry in 5–10 minutes.
  • Rate limits: freeipapi.com and urlscan.io have free-tier limits. The tool includes 300ms delays between IP lookups to respect them.
  • Revalidation: Checks are passive (DNS / IP / ASN). Frequency is per-domain in hours; the scheduler runs on demand or on a fixed cadence when enabled in SETTINGS. Decay alerts surface NXDOMAIN, IP/ASN drift, and reactivation.
  • No active probing: CRUCIBLE never sends exploit payloads, port scans, or authenticated requests. All data comes from passive sources and the live domain's public HTTP interface.

Certificate Transparency Sources
CRUCIBLE queries these in order and uses the first that returns results, so a crt.sh outage no longer breaks discovery. Cert Spotter is the primary source (free, reliable); crt.sh is the fallback. Keys for Cert Spotter / Censys come from .env — see the API Keys panel above.
Additional CT Providers (pluggable)
scantower.io doesn't publish a verified CT query API, so it's configured via .env: set both the key and a request URL template containing {q} (replaced with the domain/brand). CRUCIBLE GETs that URL with Authorization: Bearer <key> and extracts domain-like names from the JSON. The provider activates only when both SCANTOWER_API_KEY and SCANTOWER_CT_URL env vars are set AND the toggle below is on.
